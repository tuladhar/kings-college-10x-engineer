# Task API — Specification

A small REST API for managing tasks (a to-do / task-management backend).
This file is the **single source of truth** for the build. Hand it to your AI,
let it draft a plan, review the plan, then build.

## Role

You are a **senior software engineer** building a production-quality REST API.
Work to these standards:

- Write clean, idiomatic, well-structured code for the chosen language.
- Validate every input and handle errors gracefully — never trust the client.
- Write tests for the behaviour you implement.
- Prefer clarity and maintainability over cleverness.
- Follow this spec exactly. If something is ambiguous or missing, **ask before
  assuming** rather than inventing requirements.

## Goal

Expose CRUD endpoints for a `Task` resource over HTTP, returning JSON.
Keep it simple: an in-memory store is fine for the workshop — no database.

## Data model

`Task`:

| Field         | Type                              | Rules                                   |
| ------------- | --------------------------------- | --------------------------------------- |
| `id`          | string (uuid)                     | generated by the server                 |
| `title`       | string                            | **required**, 1–120 chars               |
| `description` | string                            | optional                                |
| `status`      | enum: `todo` `in-progress` `done` | defaults to `todo`                      |
| `priority`    | enum: `low` `medium` `high`       | defaults to `medium`                    |
| `dueDate`     | string (ISO 8601 date)            | optional, must be a valid date          |
| `createdAt`   | string (ISO 8601 datetime)        | generated                               |
| `updatedAt`   | string (ISO 8601 datetime)        | generated, updated on every change      |

## Endpoints

| Method   | Path         | Description                                  |
| -------- | ------------ | -------------------------------------------- |
| `POST`   | `/tasks`     | Create a task                                |
| `GET`    | `/tasks`     | List tasks (filter by `status`, `priority`)  |
| `GET`    | `/tasks/:id` | Get a single task                            |
| `PATCH`  | `/tasks/:id` | Update one or more fields of a task          |
| `DELETE` | `/tasks/:id` | Delete a task                                |

## "Done" means

- Valid JSON in and out.
- **Validation:** `title` is required; `status` and `priority` must be valid
  enum values; `dueDate` (if present) must be a valid date.
- **Status codes:** `201` on create, `200` on read/update, `204` on delete,
  `400` on validation error (with a clear message), `404` when a task isn't found.
- List endpoint supports optional `?status=` and `?priority=` query filters.

## Tests (minimum)

- **create:** success case + validation error (missing `title`).
- **update:** success case + `404` for an unknown id.
- **list:** returns an array; respects a `status` filter.

## Out of scope (keep it tight)

Auth, a real database, pagination, and rate limiting — not today.

## Deliverables

1. Working API matching this spec.
2. Unit tests for the cases above.
3. A `README.md`: setup steps, API reference, and example requests.
